**State machine BUILT — `scripts/recovery_promotion.py` (GPT re-review #4 M3):** the per-family lifecycle below is implemented as `PromotionCoordinator` driven by a total `recovery_action(state, live?, incoming?, tomb?)` table; the write-ahead `PromotionJournal` fsync's each transition to the E: volume BEFORE its mutation; only same-volume renames (live→tombstone, incoming→live) are trusted atomic, the cross-volume C:→incoming copy is re-runnable + manifest-verified and walked NO-FOLLOW (reuses Foundation 1). `assert_no_active_recovery` is the consumer fail-closed hook (the `RECOVERY_IN_PROGRESS` sentinel); the frozen family list refuses duplicates. Cross-volume copies are never atomic. The sequence (journaled per family in the run ledger):

**Quiescence contract (GPT re-review #3 M3):** promotion is unsafe unless every raw consumer is provably stopped for the WHOLE of its operation, not just at entry. Therefore: (i) every raw reader / daily job / monthly bump / builder acquires a **generation barrier (shared)** for its entire operation and, **after acquiring it, re-checks the `RECOVERY_IN_PROGRESS` sentinel** and fails closed if present (an entry-only check races a consumer that started before the sentinel or that waited on the lock and proceeded after); (ii) promotion takes the barrier **EXCLUSIVE first**, so no shared holder is mid-operation; (iii) the family set to swap is a **frozen disjoint list** fixed before the first move.

Per-family state machine with WRITE-AHEAD intents (journal fsync'd BEFORE each mutation; each transition names the exact expected on-disk state so resume is deterministic):

`COPYING -> COPY_VERIFIED -> MOVE_OLD_INTENT -> OLD_MOVED|OLD_ABSENT -> INSTALL_NEW_INTENT -> NEW_INSTALLED -> LIVE_VERIFIED -> SWAPPED`

- `OLD_ABSENT`: several incident targets are ALREADY empty — the "rename old live -> tombstone" step must accept a legitimately-absent old family (skip the move, record OLD_ABSENT) rather than assume it exists.
- Write-ahead `MOVE_OLD_INTENT`/`INSTALL_NEW_INTENT` are journaled BEFORE their renames, so a crash mid-rename is recoverable from the intent + the observed path/hash state (not from a post-hoc `OLD_MOVED` that a crash could precede).
- **Owned resume of existing incoming/tombstone paths**: a pre-existing `.recovery_incoming`/tombstone is refused UNLESS the run ledger's own intent records own it (this run created it) AND its hash matches — that reconciles "refuse pre-existing destinations" (a foreign collision) with "legitimate crash resume" (our own half-done move).
- A full **state x path x hash recovery table** maps every (journal state, live-path present?, incoming present?, tombstone present?, hashes) tuple to a single deterministic roll-forward or rollback action; resume consults it, never guesses.

2. Write a durable E:-side **`RECOVERY_IN_PROGRESS` sentinel** BEFORE the first swap — all raw readers, daily/monthly jobs, and builders **fail closed while it exists** (checked at their entry points; wired before promotion is authorized).
